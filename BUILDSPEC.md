# Design: Job Radar — Matching Brain on Borrowed Discovery

## Problem Statement

Ankita is running an active PM job search. Relevant Senior/Principal/Staff/Technical
PM roles are scattered across hundreds of sources, duplicated across boards, and
existing alerts match on keywords instead of her actual background. She wrote a full
32-section PRD ("Personalized Job Radar") for a single-user system that discovers,
dedupes, scores, and explains PM openings against her resume and preferences.

The PRD's own hypothesis (§31): the hard part is not collecting jobs, it is
consistently identifying the small number worth her attention. This design cuts the
PRD to the scope that tests that hypothesis during her actual search window.

## What Makes This Cool

A matcher that reads *her* resume and says, per job: "this Staff PM role matches your
post-op-triage healthtech work and platform experience; concern: they want 3 years of
marketplace experience; recommendation: strongly review" — with evidence cited from
both the posting and her profile. That does not exist in any job tool today.
Plus: the radar tunes itself from plain-English feedback ("dismiss the Databricks
one, too much travel") by proposing edits to its own preferences file — the PRD's
Phase 2 "natural-language preference editing" delivered on day 1.

## Constraints

- Must deliver value inside the current job search window — days, not months.
- Single user, runs locally on her Windows machine. No auth, no multi-tenancy.
- Compliance posture from PRD §11.4: public JSON APIs and permitted feeds for all
  Tier-1 sources; no LinkedIn scraping, no anti-bot evasion, no auto-apply (PRD
  non-goals §5 stand). One explicit, labeled exception: JobSpy's Indeed/
  ZipRecruiter/Google Jobs modes are best-effort scraping of public listing pages
  (no login, no captcha/anti-bot evasion). JobSpy is a supplementary source only —
  if it gets blocked, it dies gracefully and the radar runs on ATS sources alone.
- LLM cost control: dedupe and hard-filter BEFORE any LLM call; cache evaluations
  by content hash (PRD §23.6).
- Doubles as a portfolio artifact for agentic-AI PM interviews: clean repo, README
  with the "wrote the PRD, then cut 60% of it" narrative, screenshots.

## Premises

1. The differentiating value is personalized scoring + explanations, not source
   breadth. Discovery is commoditized: Greenhouse (`boards-api.greenhouse.io/v1/boards/{slug}/jobs`),
   Lever (`api.lever.co/v0/postings/{slug}`), and Ashby all expose public no-auth
   JSON per company; JobSpy covers major aggregators; hiring.cafe already dedupes
   ATS-direct listings. Build the matcher, borrow the discovery.
   — implicitly accepted via approach choice.
2. Precision beats recall. Success criteria (≤10 min/day review, ≥5 strong
   matches/week, 20% save rate) mean a missed job is invisible but a bad
   recommendation erodes trust immediately. Tune for fewer, better cards.
3. Feedback learning = a human-approved diff to preferences.yaml, proposed by the
   LLM from dismissal reasons and notes. No trained model, no silent weight drift
   (satisfies PRD §16.4 inspectability by construction).
4. The v1 dashboard is one ranked feed + save/dismiss + detail expander. The PRD
   §17's six sections, filters, and sorting wait until the two-week trial proves
   the matcher.
5. The two-week trial (PRD §26) is the real spec. Digest email, applied-tracking,
   compensation normalization, and source-health dashboards are all post-trial.

Kept from the PRD unchanged (these are right): hard filters separate from ranking
(§15.2/15.3), never invent missing fields — label unknown (§13.2), inferred vs
stated distinguishable, dedupe-before-LLM cost ordering (§23.6), tiers +
explanations emphasized over raw score precision (§29.8).

## Approaches Considered

### Approach A: Matching Brain on Borrowed Discovery (CHOSEN)
Skip building scrapers. Ingest from Greenhouse/Lever/Ashby public JSON APIs over a
curated ~50-company slug list, plus JobSpy for aggregator coverage. Spend all real
effort on the scorer: deterministic hard filters in Python, then Claude structured
output producing dimension scores, evidence-cited reasons, concerns, and confidence
against resume + preferences.yaml. SQLite. Streamlit ranked feed with save/dismiss.
Windows Task Scheduler daily run.
Effort: S-M (CC: ~2-4 sessions). Risk: Low.

### Approach B: The Full PRD Architecture
Source adapter registry, scheduler, normalization pipeline, near-dup detection,
Postgres, React dashboard, source observability, digest. Effort: XL. Risk: High —
the radar finishes after the job search does; 80% of effort lands on commoditized
plumbing (the PRD's own risk §29.7).

### Approach C: The Radar IS an Agent
No web app. Scheduled Agent SDK job writes a daily HTML/markdown brief;
conversational save/dismiss edits its own preferences. Effort: S. Risk: Med —
loses the structured disposition data the learning loop needs.

## Recommended Approach

**A, with C's feedback mechanism grafted on.**

### Architecture (v1)

```
[Task Scheduler, daily]
      │
      ▼
 fetch.py ──► Greenhouse/Lever/Ashby JSON (companies.yaml slug list)  [primary]
      │   └─► JobSpy (Indeed, ZipRecruiter, Google Jobs)  [supplementary,
      │        may die when blocked — run must succeed without it]
      ▼
 normalize.py     → common Job schema (subset of PRD §21.4; unknown stays unknown)
                    Per-ATS rules (eng review D3):
                    - Greenhouse: `content` is HTML-ESCAPED HTML — html.unescape
                      then strip tags, or dedup hashes and LLM inputs are poisoned
                    - Lever: description = descriptionPlain + lists[] assembled
                    - Ashby: GET /posting-api/job-board/{slug}?includeCompensation=true
                      (POST returns 401 — verified live 2026-07-16); response has
                      descriptionPlain, isRemote, compensation fields directly
                    - Remote status rule chain: structured field → title/location
                      regex → LLM fallback, each result labeled stated|inferred
                      (PRD §13.2)
      ▼
 dedupe.py        → exact: content_hash + (normalized company, title, location) key
                    near: rapidfuzz description similarity ≥ 90 (config, not code),
                    blocked by normalized company — compare new listings only
                    against active listings at the same company (eng review D6)
                    canonical source preference: ATS-direct > aggregator (PRD §14.2)
      ▼
 filters.py       → deterministic hard filters (PRD §15.2): seniority exclusions,
                    US eligibility, employment type, salary floor if stated,
                    visa sponsorship (reject only EXPLICIT no-sponsorship /
                    citizens-only / clearance / ITAR language — silence passes
                    and the scorer flags sponsorship as unconfirmed).
                    Rejected rows kept in DB with filter_reason (inspectable).
      ▼
 score.py         → Claude (structured output), ONLY for new/changed listings that
                    passed filters. Input: normalized job + profile.md +
                    preferences.yaml. LLM output: dimension_scores (0-10 each,
                    evidence-cited), reasons[2-4], concerns[0-3], confidence,
                    recommendation. Python then computes overall = weighted sum
                    (weights.json) and derives tier from score bands defined in
                    weights.json — the LLM never assigns overall or tier.
                    Cache key: (content_hash, prompt_version,
                    hash(preferences.yaml), hash(profile.md)). weights.json
                    changes need NO LLM re-call — only a Python recompute of
                    overall/tier over stored dimension_scores; that distinction
                    is the cost-control win.
                    Cost controls (eng review D2): the shared prefix
                    (profile.md + preferences.yaml + rubric) is marked for
                    Anthropic prompt caching — the scan is a burst of calls,
                    the ideal caching shape. max_llm_calls_per_run in
                    weights.json caps each run (freshest postings scored
                    first); first run scores only postings ≤30 days old.
      ▼
 radar.db (SQLite, WAL mode) → raw_listings, jobs, evaluations, dispositions,
                    scan_runs; single scan-in-progress lock so the manual-scan
                    button and the scheduled run never write concurrently
      ▼
 app.py (Streamlit) → ranked feed (Excellent/Strong emphasized), job cards with
                    reasons/concerns/score breakdown, Save/Dismiss(+reason chips),
                    canonical link, detail expander, "last scan" header line,
                    and a Tune box (free text → LLM proposes preferences.yaml diff
                    → user approves → file updated. Rescoring semantics: the
                    preferences hash in the cache key changes, invalidating cached
                    evaluations for all ACTIVE jobs (non-dismissed,
                    non-possibly_removed); they rescore on next scan or via the
                    manual-scan button. Dismissed jobs stay dismissed.)
```

### Key decisions

- **Score composition is deterministic.** The LLM outputs per-dimension 0-10
  ratings + evidence; Python multiplies by weights.json, sums, and maps the total
  to a tier via bands defined in weights.json. Weights are normalized so overall
  is 0-100; bands are half-open intervals (defaults: Excellent [85, 100],
  Strong [75, 85), Possible [60, 75), Weak [40, 60), hidden <40 — the PRD
  §15.5 ladder). Same
  rules version → reproducible (PRD FR-6). Weights and bands editable without
  code changes.
- **Profile = two files, human-owned.** `profile.md` (extracted from resume, she
  edits it directly — no profile editor UI) and `preferences.yaml` (PRD §10.4
  structure as written). These are the authoritative sources per PRD §6.2.
  Editing either file changes its hash in the evaluation cache key, so active
  jobs rescore automatically on the next run — same mechanism as the Tune box.
- **companies.yaml is the Tier-1 source registry.** ~50 target companies with
  ATS type + slug. Adding a company = one YAML line. This replaces the PRD §11.2
  registry table for v1.
- **Windows-native ops.** Task Scheduler for the daily scan, configured with
  "run task as soon as possible after a scheduled start is missed" (laptop may be
  asleep at scan time). Manual scan = a button in Streamlit that shells the
  pipeline; the button is disabled while the scan-in-progress lock is held.
  Scan runs logged to scan_runs (started, completed, sources attempted/failed,
  new/changed counts — PRD §12.3 subset). One source failing must not kill the
  run (try/except per source).
- **Freshness/repost logic v1:** last_seen_at updated when a listing reappears;
  a job absent from its source for 3 consecutive scans (config, not code) →
  status possibly_removed. Source-failure semantics (eng review D9): absence
  increments ONLY when that source's scan succeeded — a failed fetch never
  counts as a job disappearing. If zero sources succeed, the scan is marked
  status=failed and the dashboard shows a failure banner instead of an
  innocently empty feed. Full §19 verification deferred.

## Test Strategy (eng review D5)

Full pytest suite written alongside the code, not deferred — 23 traced paths:
- **Fixtures:** one recorded JSON payload per ATS (Greenhouse/Lever/Ashby) checked
  into tests/fixtures/; normalization tests pin the D3 rules (HTML unescape,
  Lever assembly, Ashby salary, stated-vs-inferred remote labels).
- **Unit:** dedupe (exact, near ≥90 merges / <90 doesn't, canonical preference),
  filters (seniority/salary-floor/unknown-passes, filter_reason retained),
  scoring math (weighted sum, half-open band edges: 84.9→Strong, 85.0→Excellent),
  cache (hit skips LLM, preferences-hash change invalidates, dismissed stay
  dismissed), cap (max_llm_calls_per_run honored, freshest first), malformed
  LLM output → retry then skip-with-log.
- **E2E:** fixtures → normalize → dedupe → filter → score (stubbed LLM) → SQLite
  → feed query, plus the scan-lock concurrency path.
- **Eval:** golden_set.jsonl ≥12/15 tier agreement (the centerpiece, step 3 of
  Next Steps).

## Success Criteria

The PRD §26 two-week trial, unchanged:
- ≥5 strong/excellent matches per week (market permitting)
- ≥1 valuable role she had not already seen elsewhere
- Daily review ≤10 minutes; she prefers the radar to manual board-checking
- ≥20% of excellent/strong matches saved or applied
- Explanations judged accurate (spot-check: no generic "your skills are a good fit")
- Matcher eval: on the 15-job golden set, tier assignment agrees with her hand
  labels on ≥12/15, with zero "obviously wrong" cards in the top tier

## Next Steps

1. **Golden set first (the assignment):** hand-pick 15 real postings (5 clearly
   great, 5 borderline, 5 wrong-for-her), transcribe each into the normalized Job
   schema as `golden_set.jsonl`, and label each with expected tier + one-line
   reason. This is the matcher eval before any ingestion code exists — score.py
   consumes golden_set.jsonl directly, no fetcher required.
2. Draft profile.md from resume + preferences.yaml (PRD §10.4 shape) + weights.json.
3. Build score.py + prompt + a ~30-line eval script that runs it over
   golden_set.jsonl and reports tier agreement vs. labels; iterate until ≥12/15.
4. Build fetch.py (Greenhouse/Lever/Ashby + JobSpy) + normalize + dedupe + filters.
5. Wire SQLite + Streamlit feed + save/dismiss + Tune box.
6. Schedule daily scan; start the two-week trial; log every false positive.
