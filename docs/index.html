The product decision that shaped everything

The original PRD had 32 sections — and its own hypothesis disagreed with its scope.

Fifteen of those sections specced ingestion plumbing: scrapers, source registries, adapters. But the PRD's stated hypothesis said the hard part was never collecting jobs — it was identifying the few worth a candidate's attention. Investigation confirmed discovery is commoditized: Greenhouse, Lever, and Ashby all expose public JSON APIs, and aggregators cover the rest. So the build inverted the ratio — borrow the discovery, build the matching brain — and roughly 60% of the PRD was deliberately cut or deferred behind explicit un-defer triggers.

The split of responsibilities is the design's core idea: the LLM judges fit — with evidence quoted from both the posting and the profile — while Python owns everything that must be reproducible: the weighted score, tier bands, preference boosts, location caps, and every hard filter. Rules the candidate states explicitly ("this company is a direct target") live in config-driven code, not in prompt language, because deterministic rules apply 100% of the time and LLM judgment doesn't.

Calibrated against a human, honestly

Before trusting the matcher, its owner hand-labeled 22 real postings — great, borderline, and deliberately wrong — and the scorer was tuned until its tiers tracked hers. The measured result: 21/22 within one tier, every hard-reject correctly suppressed, and 11 of 13 apply-worthy roles in the primary feed. The honest finding along the way: single-pass LLM scoring carries ±5–8 points of run-to-run variance, so taste rules discovered during calibration (dream-company boosts, a location cap, senior-IC-only scope) were promoted from prompt guidance into deterministic code, where they hold every time.

Product guardrails

  • Never invents data — unknown salary stays unknown, inferred fields are labeled inferred
  • Never auto-applies, contacts anyone, or acts on the candidate's behalf
  • Rejects on visa sponsorship only when a posting explicitly excludes it — silence is flagged, not punished
  • A failed source never counts as a disappeared job; a fully failed scan is loud, never an innocently empty feed
  • Every LLM evaluation is cached by content + prompt + preference version — cost is a dial, not a surprise
  • Feedback loop: dismissals with reasons and plain-English tuning edit the preference file through a human-approved diff

Stack & scale notes

Python pipeline on a daily scheduler; SQLite in WAL mode; FastAPI + HTMX dashboard with tiered card density (top matches expanded with evidence, the long tail collapsed to scannable rows); Claude with prompt caching for scoring bursts; pytest suite covering normalization quirks, dedup, filter edges, scoring math, cache invalidation, and pipeline failure semantics.
